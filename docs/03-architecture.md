# Архитектура truemail

truemail состоит из Rust-ядра и тонкого Tauri-интерфейса. Ядро владеет
аккаунтами, синхронизацией, локальной SQLCipher-базой, зашифрованными blob-файлами,
поиском и локальным API. UI вызывает команды ядра и не реализует транспортную
логику самостоятельно.

## Хранилище и миграции

Канонический запуск миграций — `Db::migrate`. Он сначала применяет SQL-файлы,
затем выполняет обязательные шаги приложения: шифрует старые значения настроек,
очищает свободные страницы после миграции `0009` и импортирует старые правила.
Поэтому один `sqlx migrate run` создаёт только структурно обновлённую базу и не
заменяет запуск `Db::migrate`. Этот контракт проверяет тест
`sqlx_migrations_are_completed_by_the_application_contract`.

Raw RFC 5322/MIME хранится в зашифрованном blob-store. Разобранное тело и
нормализованные метаданные вложений кэшируются в `message_content_cache`; ключом
версии служит `raw_blob_ref`, а raw остаётся источником истины.

## Поиск

`SearchIndex` — единственная точка записи тела и выполнения полнотекстового
поиска. Текущая реализация использует SQLite FTS5, но может быть заменена без
изменения импорта писем.

В виртуальной таблице `messages_fts` колонка `message_id` — это локальный
числовой `messages.id`, а не заголовок RFC `Message-ID`. Канонический идентификатор
RFC хранится отдельно в `messages.rfc822_message_id`. Колонка FTS сохранена под
старым именем ради неизменности уже применённой миграции `0008`; для связей код
использует `rowid`.

## Gmail push в desktop-only архитектуре

truemail не содержит облачного сервера и не передаёт токены или почтовые данные
своей инфраструктуре. Поэтому Gmail `users.watch` намеренно не используется:
Google доставляет такие события только через Cloud Pub/Sub, а подписку надо
создавать в Google Cloud, принимать и подтверждать на сервере и продлевать не
реже одного раза в семь дней. Это превратило бы локальное приложение в сервис с
постоянной внешней инфраструктурой.

Для установленного desktop-клиента выбран рекомендованный Google polling:
каждые 25 секунд запрашиваются только ID последних входящих. Новый ID запускает
инкрементальную синхронизацию по сохранённому `historyId`; полный фоновый проход
раз в пять минут подхватывает изменения без нового входящего письма. Таким
образом, обычное новое письмо появляется почти сразу, а приложение остаётся
самодостаточным и продолжает работать без truemail-сервера.
