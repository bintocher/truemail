# Что ещё нужно реализовать в truemail

Дата повторной сверки: 2026-07-17.

В файле перечислены только подтверждённые пробелы текущей версии. Уже
реализованные пункты предыдущего аудита удалены.

## 1. Почта и провайдеры

### Поддержка провайдеров

- Для Outlook/Exchange Online пока нет рабочего OAuth-подключения.
- JMAP остаётся только вариантом модели данных: рабочего транспорта и
  диспетчеризации для него нет.

### Gmail push

- Почти реальное время Gmail сделано коротким polling последних ID. Для
  полноценного push всё ещё нужен отдельный механизм Gmail watch/Pub/Sub либо
  явно зафиксированное ограничение desktop-only архитектуры.

### Ещё отсутствующие почтовые функции

- Snooze входящих писем не реализован. Колонка `messages.snoozed_until` есть,
  но команд, UI и фонового пробуждения нет.
- Нет шаблонов писем.
- Нет редактора и автоматической подстановки подписей. Таблица `signatures`
  и CSS редактора существуют, а рабочей настройки нет; текст раздела аккаунтов
  при этом всё ещё обещает подпись.
- Есть просмотр и копирование исходного MIME, но нет экспорта письма в `.eml`.
- Нет свайпов и drag-and-drop писем между папками.
- Нет автообновления приложения и защиты от запуска нескольких экземпляров
  (`tauri-plugin-updater` и `tauri-plugin-single-instance` не подключены).

## 2. Синхронизация, правила и фоновые операции

- Умные папки вычисляются во фронтенде и сохраняются одной JSON-настройкой
  `smart_folders_ui`. Таблицы `smart_folders` / `smart_conditions` не являются
  источником истины, а core не умеет выполнять условия.
- Настройка сквозных папок хранится набором ключей `unified_<folder_id>`, тогда
  как таблицы `unified_folders` / `unified_sources` не используются.
- Защита параллельного sync находится в Tauri-слое (`syncing`/`syncing_aux`).
  Сам `AccountManager` не запрещает двум другим клиентам core одновременно
  синхронизировать один аккаунт.

## 3. Календарь и контакты

- Редактор события не позволяет задавать повторение, исключения, участников,
  организатора, напоминания и другие уже предусмотренные моделью экспертные
  поля.
- Перетаскивание события на другую дату/время отсутствует. Изменение даты
  возможно только через редактор.

## 4. Внешний API

- MCP/REST-сервера нет. `crates/core/src/api/mod.rs` содержит только модели
  capability и статический список описаний инструментов.
- Нет управления API-клиентами и токенами, сетевого listener, вызова
  инструментов, capability-проверки на реальных операциях и записи аудита.
- Таблицы `api_clients` и `api_audit` не используются.

## 5. Безопасность и восстановление

- Постоянные ключи затираются через `zeroize`, но OAuth credential и токены
  остаются обычными `String` и не затираются после использования.

## 6. Производительность и хранение

- В списке писем есть курсорная подгрузка по 100 записей, но нет
  виртуализации: все загруженные строки остаются DOM-узлами. При долгой
  прокрутке большой папки DOM продолжает расти.
- `get_message` при каждом открытии заново читает и парсит raw MIME, включая
  HTML и вложения. Нет кэша разобранного тела или нормализованных MIME-частей.
- `SearchIndex` остаётся частично декоративной абстракцией: импорт писем пишет
  тело напрямую в `messages_fts`, обходя `SearchIndex::index_body`.

## 7. Интерфейс и UX

- Активный фильтр списка сбрасывается после курсорной подгрузки при скролле:
  `loadNextMessagePage` повторно вызывает `renderMessageList` с полной папкой,
  минуя `applyListOptions`. Фильтр и сортировка должны применяться и к новым
  страницам без сброса позиции прокрутки.
- Нет `Ctrl+A` для выделения всех писем текущего списка. Кнопка «Выбрать все»
  есть, но стандартное сочетание клавиш не обрабатывается.
- Массовое удаление выполняется сразу. При удалении более 10 выбранных писем
  нужно показывать подтверждение с точным количеством; для 10 и менее оставить
  быстрое действие и существующий undo.
- Первый запуск по-прежнему показывает приветствие только после языка, выбора
  папки и создания ключей. Приветствие и объяснение процесса должны идти до
  создания хранилища.
- Для Яндекса OAuth всё ещё требует ручного копирования кода. Loopback callback
  реализован только для Gmail.
- Текст раздела тем обещает импорт тем, но UI и формат импорта отсутствуют.
- Горячие клавиши теперь честно показаны как фиксированные рабочие сочетания,
  но переназначать их нельзя; таблица `keybindings` не используется.
- Навигация всё ещё построена преимущественно на `div`. JavaScript добавляет
  `role`, `tabindex` и обработку Enter/Space во время выполнения, но нужен
  переход на семантические `button`/`a` и полноценная проверка экранным
  диктором.
- Локализация UI и core разделена между большим JS-словарём и Fluent-файлами.
  Нужен единый источник переводов и автоматическая проверка отсутствующих
  ключей/захардкоженных строк.
- `app.js` остаётся монолитом более чем на тысячу строк. Почту, композер,
  календарь, контакты, правила, настройки и i18n нужно разнести по модулям.

## 8. Схема и документация

- Не используются либо используются обходным путём таблицы `signatures`,
  `keybindings`, `image_trust`, `api_clients`, `api_audit`,
  `unified_folders` и `unified_sources`.
  Нужно либо подключить их как источник истины, либо удалить из схемы.
- `messages_fts.message_id` фактически содержит числовой локальный PK/rowid, а
  имя похоже на RFC Message-ID. Поле стоит переименовать или явно
  задокументировать.
- Миграция шифрования настроек `0009` завершается кодом приложения. Отдельный
  прогон миграций через `sqlx-cli` не создаёт полностью готовое состояние;
  этот контракт нужно формализовать и проверять отдельным тестом.
- В комментариях исходников остаются ссылки на отсутствующие `docs/05-*`,
  `docs/06-*` и `docs/10-*`. Нужно добавить документы либо убрать ссылки.

## 9. Тесты и сборка

- Есть CI и 37 unit/integration тестов, но основные сценарии outbox/retry, CRUD
  календаря/контактов и команды Tauri почти не покрыты.
- Нет сквозного теста: подключение тестового аккаунта -> sync -> чтение HTML и
  вложений -> отправка -> серверное перемещение/undo.
- На Linux/macOS `Makefile` использует shell-скрипт, но на Windows всё ещё
  требуется отдельно установленный `make`, которого нет в стандартной системе.
  Нужен нативный bootstrap-командлет/скрипт без обязательного GNU Make либо
  документированная установка инструмента.
- CI запускается только на `self-hosted, linux, x64`. Нет Windows-раннера,
  который проверяет сборку Tauri, Windows-specific ветки, keyring, tray,
  autostart и упаковку `.msi`/`.exe`.

## Приоритет

### P2

1. Внешний MCP/REST API с capability и аудитом.
2. Виртуализация списка писем и сохранение фильтра при подгрузке.
3. `Ctrl+A` и подтверждение удаления более 10 писем.
4. Snooze, подписи, шаблоны и экспорт `.eml`.
5. Семантическая доступность и единая локализация.
6. Модульная декомпозиция UI и расширение тестового покрытия.

### P3

7. Автообновление и single-instance.
8. Редактирование расширенных полей календаря и drag-and-drop событий.
9. Удаление или подключение оставшихся мёртвых таблиц и устаревших ссылок
    на документацию.
10. Windows-раннер для CI и проверки Windows-сборки.
