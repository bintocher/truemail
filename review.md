# Оценка проекта truemail

Дата аудита: 2026-07-14. Охват: crates/core, apps/desktop (src-tauri + ui), миграции БД, репозиторий.
Метод: 4 независимых проверки (ядро Rust, Tauri/IPC, UI/UX, схема БД и гигиена репозитория).

## Резюме

Сейчас это read-only клиент Яндекс-почты с хорошим фундаментом шифрования и
недоделанным всем остальным. Письма скачиваются и читаются (только plain text),
календарь и контакты выгружаются. Отправки писем нет вообще. Других провайдеров
подключить нельзя, несмотря на заявленные 6. Значительная часть UI - плацебо:
кнопки и тумблеры, которые ничего не делают. Есть 2 критичных бага, портящих
данные пользователя, 3 критичных проблемы безопасности и 4 критичных проблемы
репозитория. Ниже - полный разбор.

Сильные стороны, чтобы быть честным:

- SQLCipher-шифрование БД целиком + ключи в системном keychain + миграция
  plaintext -> SQLCipher с бэкапом, verify и затиранием исходника (storage/mod.rs:194-352).
  Покрыто интеграционными тестами. Это сделано серьезно.
- OAuth PKCE без client_secret, токены в keychain, а не в БД. Пароль через
  приложение не проходит. Секреты в логи не попадают.
- TLS без фоллбеков: rustls + webpki-roots, полная валидация (imap.rs:72-86).
- Темы (светлая/темная/системная/акценты/плотность) реально работают.
- Тексты-пояснения в настройках - лучшее место продукта.
- Пустые состояния с CTA, aria-live в визарде, обработка истекшего OAuth-кода.

## 1. Что есть (реально работает)

### Ядро (crates/core)

- Bootstrap: SQLCipher-БД, миграции, FTS5-индекс (lib.rs:33-50).
- Шифрование: SQLCipher для всей БД (storage/mod.rs:29-56), ChaCha20-Poly1305
  для blob-store (blobs.rs:22-50) и настроек (storage/mod.rs:70-157), ключи в keychain.
- IMAP (только Yandex, только XOAUTH2): TLS, LIST c SPECIAL-USE, STATUS,
  UID FETCH последних 500 писем на папку (imap.rs:61-282). Свой декодер
  modified UTF-7 с тестами.
- CalDAV/CardDAV Yandex: discovery, PROPFIND, REPORT, парсинг VEVENT/vCard (dav.rs:119-437).
- OAuth2 Yandex: PKCE S256, обмен кода на токен (oauth.rs:48-134).
- Хранение: аккаунты, папки, письма (парсинг mail-parser + FTS), контакты,
  события (storage/repo.rs).
- Поиск FTS5 + раскладко-независимые варианты запроса RU/EN (search/mod.rs:51-65).
- i18n на Fluent для ядра (i18n/mod.rs).

### Tauri-слой

- Все 20 команд invoke_handler реальные, фиктивных нет (main.rs:116-138).
- initialize_storage с откатом при ошибке (удаление ключей и файлов БД, commands.rs:151-170).
- storage_status / move_storage / clear_local_data / open_data_dir.
- begin_account_connection / complete_yandex_oauth + фоновый sync.

### UI

- Визард первого запуска, 8 шагов, реальный вызов ядра, OAuth через bridge.
- Список писем из ядра, фильтры и сортировка, чтение письма (plain text), markSeen.
- Поиск через палитру (Ctrl+K) с защитой от гонок и RU/EN раскладкой.
- Умные папки: 8 встроенных + пользовательский редактор условий И/ИЛИ с персистом.
- Настройка тулбара письма (порядок, видимость, подписи) с персистом.
- Календарь: месяц/неделя/день, реальные события, drag&drop по ячейкам (только DOM).
- Настройки: 11 разделов; темы, плотность, акцент, масштаб, хранилище - работают.
- Контекстные меню, ресайз панели, drag&drop файлов в композер (визуально).

## 2. Чего нет (заглушки и отсутствующее)

### Фундаментальное

- ОТПРАВКА ПИСЕМ. Нет SMTP-клиента в зависимостях (ни lettre, ничего). Кнопка
  "Отправить" в композере без обработчика (index.html:292). Скоуп mail:smtp
  запрашивается, smtp_host пишется в БД - но отправить письмо нечем. Это не
  почтовый клиент, пока этого нет.
- HTML-письма не отображаются: рендер только body_text через textContent
  (app.js:498). В 2026 это неюзабельно - большинство писем HTML.
- Провайдеры кроме Yandex: подключить невозможно. autoconfig знает 5 доменов,
  но есть только connect_yandex с захардкоженным imap.yandex.com (imap.rs:62).
  Парольная/app-password аутентификация не реализована вообще.
- EWS, JMAP, NTLM: только enum-значения в модели, ни строчки транспорта.
- Инкрементальный sync: нет. Колонки uidvalidity/uidnext/highestmodseq в схеме
  есть, нигде не используются. Каждый sync - полная перекачка до 500 RFC822 на
  папку. На большом ящике это гигабайты при каждом запуске.
- Каталог src/sync/ (заявлен в README) не существует.
- Outbox-воркер: enqueue_op пишет в outbox_ops (repo.rs:402-414), но обработчика
  нет нигде. "Прочитано" не уходит на сервер и молча копится в таблице.
- Refresh OAuth-токена: refresh_token сохраняется, но не используется. После
  истечения access_token синхронизация ломается навсегда без ре-логина.
- Внешний API / MCP / REST: только статический список описаний инструментов
  (api/mod.rs). Сервера нет, аудита нет, таблицы api_* мертвые.
- Периодическая синхронизация: sync один раз на старте + 3 reload по таймерам
  (bridge.js:85). Дальше почта не обновляется до перезапуска. IDLE нет.

### Недоделанное в ядре

- Треды: таблицы и колонки есть, thread_id всегда NULL, JWZ-алгоритма нет.
- Вложения: has_attachments считается, но таблица attachments не заполняется,
  get_message всегда возвращает пустой список (repo.rs:358).
- Отписка от рассылок: is_newsletter детектится, unsubscribe всегда None (repo.rs:360).
- DKIM/SPF/DMARC: колонки и модель есть, Authentication-Results не парсится.
- Мастер-пароль (Argon2id): заявлен в доке модуля crypto (crypto/mod.rs:7),
  argon2 в зависимостях отсутствует.
- Умные папки в core: только чтение списка, матчинга условий нет. UI считает
  их сам в JS, таблицы smart_folders/smart_conditions в БД мертвые.
- Мертвые таблицы без кода: threads, labels, attachments, api_*, unified_*,
  image_trust, signatures, event_attendees, event_alarms, keybindings.

### Плацебо в UI (кнопка есть, действия нет)

- Композер: "Отправить", "Отправить позже", "Удалить черновик", вся панель
  форматирования, кнопка вложения. Drop файла рисует чип, но файл никуда не
  прикрепляется (app.js:566-574).
- Все пункты меню "Ещё" и контекстных меню только закрывают меню (app.js:142, 303).
- Архив/Отложить в тулбаре; "Удалить" убирает строку из DOM без вызова ядра -
  после перезапуска письмо возвращается (app.js:278).
- Тумблеры настроек: автозапуск, трей, уведомления, треды, превью, приватность,
  журнал протоколов - переключаются визуально, ничего не сохраняют (app.js:216).
- Редактор события: "Сохранить" = закрыть модалку (app.js:328), "Удалить" без
  обработчика. Drag&drop события меняет только DOM, данные не сохраняются (app.js:338).
- "+ Контакт", поиск по контактам, карточка контакта - мертвые.
- Переназначение хоткеев записывается в span, не сохраняется (app.js:305-316).
- Импорт/экспорт темы, радиус, размер шрифта - декорация.
- Хардкоды: прогресс-бар диска 44/31/17/8 (styles.css:362), "~14 писем"
  (index.html:341), "Демо truemail" в форме события (index.html:390).

### Отсутствует из ожидаемого в почтовом клиенте

Snooze (кнопка и хоткей есть, функционала и колонки в БД нет), отложенная
отправка, шаблоны, редактор подписей (выпилен: CSS остался, HTML/JS нет),
правила/фильтры, печать, экспорт .eml, пометить непрочитанным, перемещение в
папку, спам, множественное выделение писем, undo, свайпы/быстрые действия,
спеллчек (принудительно отключен, index.html:275), уведомления о новых письмах
(plugin-notification отсутствует в Cargo.toml), трей, автообновление (updater),
сохранение размера/позиции окна, single-instance.

## 3. Что неправильно (баги)

### Критично - портит данные пользователя

1. imap.rs:244: `uid_fetch(set, "(UID RFC822 ...)")` - RFC822 по RFC 3501
   УСТАНАВЛИВАЕТ \Seen на сервере. Первая же синхронизация помечает все
   непрочитанные письма прочитанными во всем ящике - и в вебе, и в других
   клиентах. Нужен BODY.PEEK[].
2. Дубли при ресинке: contacts без UNIQUE(account_id, uid) (0004_contacts.sql),
   events без UNIQUE(calendar_id, uid) (0005_calendar.sql), то же calendars,
   contact_emails, event_attendees, event_alarms, threads. Сейчас спасает
   только DELETE-all-INSERT в save_yandex_dav, но это своя беда (см. ниже).
3. FTS-сироты: каскадные DELETE (аккаунт -> папки -> письма) НЕ запускают
   DELETE-триггеры FTS при выключенном PRAGMA recursive_triggers (по умолчанию
   off). Удаление аккаунта навсегда оставляет весь текст его писем в
   messages_fts - утечка данных удаленного аккаунта + распухание индекса.

### Серьезно

4. save_yandex_dav (repo.rs:112-181): DELETE всех контактов/календарей + полная
   перевставка на каждый sync - теряются локальные id; старые блобы не
   удаляются - бесконечная утечка блобов; blobs.put до транзакции - при ошибке
   INSERT остаются блобы-сироты.
5. enqueue_op пишет payload {"seen":true} без message_id/uid/folder
   (repo.rs:398-399) - даже будущий outbox-воркер не поймет, к какому письму
   относится операция. Данные очереди бесполезны.
6. Пять expect в main() (main.rs:51-140): недоступный keyring или битая БД =
   мгновенная паника, а из-за windows_subsystem="windows" пользователь не
   увидит вообще ничего - приложение молча не запустится.
7. FTS-поиск: сырой запрос уходит в MATCH (search/mod.rs:38-41) - кавычки,
   двоеточия и операторы FTS5 дают ошибку, которая глушится unwrap_or_default
   (commands.rs:226). Поиск "молча не работает".
8. move_storage (commands.rs:294-328): копирует БД, пока фоновый sync может
   писать новые blobs - потеря данных между checkpoint и переключением;
   приложение продолжает писать в старый путь до перезапуска.
9. OAuth на Windows требует установленный Яндекс Браузер, иначе подключение
   аккаунта невозможно (commands.rs:469-471). Нет fallback на системный браузер.
10. "Ответить" не заполняет "Кому" и не цитирует тело - подставляется только
    тема (app.js:88-89). Одно поле "Копия" на CC и BCC (index.html:272).
11. renderMessageList автокликает первое письмо (app.js:487) - каждая смена
    папки помечает верхнее письмо прочитанным.
12. Нет пагинации/виртуализации: 500 писем на КАЖДУЮ папку разом при старте
    (bridge.js:46), все DOM-узлы подряд. На 50к писем UI ляжет.
13. reloadCoreData по таймерам пересоздает DOM под пользователем - сбрасывается
    выделение и скролл, повторяется автоклик (bridge.js:85).
14. mark_seen: два statement без транзакции (repo.rs:392-400).
15. list_messages_by_ids - N+1 запросов в цикле (repo.rs:304-320); get_message -
    3 запроса вместо одного; письмо парсится дважды (при импорте и при каждом чтении).
16. Календарь: неделя/день всегда показывают текущую дату, стрелки листают
    только месяц (app.js:106, 224). Наложение событий одного часа в week-view
    не обрабатывается - лягут друг на друга (styles.css:256).
17. parse_event берет только первый VEVENT (dav.rs:247-263) - recurrence
    overrides теряются; vCard 2.1 c QUOTED-PRINTABLE не парсится.
18. Сравнение id `+row.dataset.messageId===message.id` (app.js:493) сломается
    на строковых id.
19. reqwest Client::new() без таймаута в OAuth (oauth.rs:97) - зависание на
    плохой сети.
20. root.style.zoom для масштаба (app.js:594) ломает getBoundingClientRect -
    тултипы и ресайзер панели промахиваются при zoom != 1.
21. location.reload() после онбординга и подключения аккаунта (app.js:373, 547) -
    мигание, потеря состояния.
22. clear_local_data("all") не чистит messages_fts и attachments (commands.rs:365-383).
23. Условия умных папок сравниваются по русским строкам ('не содержит',
    app.js:502) - смена языка сломает логику фильтров.
24. contextmenu подавлен глобально, включая поля ввода (app.js:137) - пользователь
    теряет системное "Вставить".
25. lib.rs:34: create_dir_all(...).ok() - ошибка создания data_dir глотается.

### Схема БД

- Нет составного индекса (folder_id, date DESC) под главный запрос списка
  писем - сортировка в памяти на больших папках.
- rfc822_message_id и in_reply_to без индексов - будущий трединг это full scan
  на каждое письмо.
- attachments.message_id - FK без индекса.
- Календарь: нет RECURRENCE-ID / EXDATE / RDATE - измененное вхождение
  повторяющегося события непредставимо.
- event_alarms.trigger_minutes - только "за N минут до начала", абсолютные
  триггеры RFC 5545 непредставимы.
- outbox_ops без status и next_attempt_at - нет ретраев/backoff.
- messages_fts.message_id заполняется числовым PK, а не Message-ID - имя врет.
- Гибрид 0009 + encrypt_legacy_settings в коде: миграция неполна без кода
  приложения; прогон через sqlx-cli оставит настройки в состоянии, на котором
  decrypt_setting падает. Хрупко, хоть и устойчиво к прерыванию.

## 4. Безопасность

Обещание продукта: "все зашифровано, нельзя скопировать папку на другую машину
и прочитать". По БД - выполнено (SQLCipher + ключ в keychain, есть тесты).
Но подрывается следующим:

### Критично

1. Имена файлов blob-store = FNV-1a(открытый текст) (blobs.rs:23-29, 55-62).
   Две проблемы:
   - детерминированный бесключевой хэш открытого текста: атакующий с копией
     папки может ПОДТВЕРДИТЬ наличие конкретного известного письма простым
     вычислением FNV (confirmation attack). Прямо противоречит модели угроз;
   - FNV не криптографический: подобрать коллизию тривиально - специально
     сформированное письмо молча перезапишет чужой blob (blobs.rs:28,
     безусловный fs::write). Тихая порча локального архива.
   Нужно: BLAKE3 с ключом (keyed hash) или случайные имена + маппинг в БД.
2. Мастер-ключи выводятся ТОЛЬКО из движений мыши, OS RNG принципиально не
   используется (crypto/mod.rs:103-116). Проверяется объем данных (32 КБ), а
   не энтропия; реальная энтропия координат мыши на байт мала. Отказ от
   смешивания с OsRng - необоснованное ослабление ключа. Правильно:
   HKDF(OsRng XOR mouse). Заодно: KDF - голый SHA-256 вместо HKDF.
3. App lock (пароль/PIN): SHA-256 без соли и KDF, verifier в обычной настройке,
   проверка НА ФРОНТЕНДЕ (app.js:599-602). Обходится правкой настройки или
   DevTools; PIN из 4 цифр перебирается мгновенно. Это бутафория, а не защита -
   хуже, чем отсутствие фичи, потому что создает ложное чувство безопасности.

### Серьезно

4. XSS от контента письма: результаты поиска вставляются в палитру через
   innerHTML БЕЗ экранирования темы/отправителя (app.js:196). Письмо с темой
   `<img src=x onerror=...>` исполнит JS при поиске. При withGlobalTauri:true
   (tauri.conf.json:10) инъекция получает window.__TAURI__ и все 20 команд.
   Такая же дыра: имя файла вложения при drop (app.js:567), value условия
   умной папки (app.js:236), чипы участников события (app.js:322).
5. PRAGMA key с ключом в hex попадает в текст SQL-statement (storage/mod.rs:170,
   233-239) - при включенном statement-логировании sqlx (debug) ключ SQLCipher
   утечет в лог.
6. Тела писем в FTS-индексе защищены только SQLCipher, тогда как оригиналы -
   еще и ChaCha20. Осознанный компромисс, но защита тел слабее защиты
   оригиналов; зафиксировать в модели угроз.
7. Нет AAD в AEAD: blob можно подменить другим валидным blob-ом той же
   установки (cut-and-paste), связка "ссылка в БД <-> blob" не защищена.
8. Случайный 96-бит nonce: birthday bound ~2^32 сообщений на ключ. Для
   архива на годы лучше XChaCha20-Poly1305.
9. Нет zeroize: ключи и токены не затираются в памяти.
10. Потеря keychain (переустановка ОС) = полная потеря архива. Экспорта/
    восстановления ключа нет. Пользователя об этом не предупреждают.
11. Изоляции рендера писем нет: когда появится HTML-рендер, текущая
    архитектура (один DOM, глобальный Tauri) не готова. Нужен sandboxed
    iframe без allow-scripts + санитайзер + CSP frame-src.
12. Лишние capabilities: shell:allow-open и global-shortcut:* фронтендом не
    используются (все на Rust-стороне) - убрать (capabilities/default.json).
13. secure_remove_file (перезапись нулями) на SSD/COW-ФС не гарантирует
    затирание - best-effort, не обещать большего.

### Что сделано правильно (не трогать)

- SQLCipher целиком, включая WAL; двойное шифрование настроек; checkpoint +
  VACUUM + secure_delete при миграции - редкая аккуратность.
- Токены только в keychain, PKCE без client_secret, state криптослучайный.
- TLS строгий, без plaintext-фоллбеков и danger_accept_invalid_certs.
- Секреты не логируются; энтропия обнуляется в JS и Rust.
- CSP приличный для текущего состояния; img-src 'self' data: побочно блокирует
  трекинг-пиксели.

## 5. UX

### Визард

- Шаг энтропии - барьер: 32 КБ = сотни движений мыши; на тачпаде мучение, на
  тачскрине невозможно (mousemove с isTrusted не покрывает touch). Нет
  альтернативы. Ошибка initialize_storage сбрасывает прогресс в 0% - заново
  возить мышью.
- Порядок шагов странный: язык -> папка -> ключи -> и только потом
  "Добро пожаловать". Приветствие должно быть первым.
- OAuth через ручное копирование кода - архаика; нужен loopback-редирект.
- При 100% энтропии хранилище создается автоматически - кнопка "Создать"
  декоративна, пользователь не успевает понять, что произошло.

### Письма

- Нет множественного выделения (ни Ctrl/Shift+клик, ни чекбоксов) и массовых
  операций - для почтового клиента блокер.
- Нет undo. "Удалить" врет: локально скрывает, после перезапуска письмо
  возвращается.
- Нет drag&drop писем в папки, нет быстрых действий на строке.
- J/K есть, но нет U (непрочитанное), star, чорды G+I заявлены в палитре,
  но не реализованы (app.js:179 - просто подпись).
- Автоклик первого письма при каждой смене папки + markSeen = агрессивно.

### Композер

- Нет автосохранения черновика: закрыл = потерял текст без предупреждения.
- Нет валидации адресов, автодополнения из контактов, chips получателей.

### Поиск

- Стрелки и Enter в палитре НЕ работают - навигация только мышью (переменная
  sel объявлена, обработчика нет, app.js:187). Нет фильтров (from:, has:),
  истории, подсветки совпадений.

### Обратная связь

- Нет тостов, нет индикатора синхронизации, ошибки sync - только в console.error.
  Пользователь не узнает, что почта не синхронизировалась.

### Доступность - плохо

- Почти вся навигация - div с onclick: без role, tabindex, aria. Tab-ом
  добраться до папок и настроек невозможно.
- Глобальное outline:none на инпутах (styles.css:260).
- Тумблеры без role="switch"/aria-checked. Тултипы только на hover.
- Контраст --text-3 #8b909a на белом ~3.1:1 - ниже WCAG AA, а этим цветом
  набраны превью, даты, подписи.

### Локализация - фикция

- locales/ru.ftl и en.ftl фронтендом НЕ используются (используются только
  ядром). Реальная i18n UI - самодельный словарь на ~10 строк, покрывающий
  только визард (app.js:341-350). 95% интерфейса - захардкоженный русский в
  HTML/JS. Переключение языка в настройках меняет почти ничего.

### Прочее

- Вспышка светлой темы на старте у dark-пользователей (тема применяется после
  ответа ядра, нет раннего инлайн-скрипта).
- alert()/confirm() для операций хранилища вместо родных модалок (app.js:596).
- min-width:940px без обработки меньших окон.
- Магические размеры шрифтов 12/12.5/13/13.5/15px вперемешку; самодельный
  atomic CSS (.mt-6/.ml-14/.w80).

## 6. Архитектура

- Нет абстракции транспорта: backend/mod.rs - 8 строк, три свободные функции
  *_yandex. Ни трейта, ни диспетчеризации по BackendKind. README обещает
  "трейт MailBackend + адаптеры imap/ews/jmap" - в коде этого нет.
- Слой хранилища знает о провайдере: save_yandex_account с хостами и цветом
  Яндекса прямо в SQL (repo.rs:10-47). Хранилище должно принимать Account.
- SearchIndex-трейт есть, но repo пишет в messages_fts напрямую (repo.rs:277-281) -
  трейт декоративный, миграция на Tantivy потребует править repo.
- Русские строки-сообщения зашиты в core (account/mod.rs:95-174) - мимо i18n.
- i18n ядра включает файлы вне крейта через ../../../../locales/ - крейт не
  самодостаточен.
- Параллельные sync не защищены в core (только флаг в GUI).
- Нет GC блобов и проверки битых ссылок blob <-> БД.
- app.js - один файл 627 строк без модулей, строки-простыни по 1000+ символов.
- Демо-данные в HTML + clearDemoData из bridge: если мост не поднялся,
  пользователь видит фейковые "3 аккаунта" и "148 контактов".

## 7. Репозиторий и опенсорс-гигиена

### Критично

- LICENSE - не лицензия: самодельная русская аннотация, сам файл признает
  "перед публикацией поместить сюда полный официальный текст AGPL-3.0"
  (LICENSE:15-16). Юридически репозиторий сейчас не под AGPL. Ссылка на CLA
  ведет в несуществующий docs/.
- В git ровно один коммит: закоммичены только .gitignore и README. ВЕСЬ код
  (crates/, apps/, миграции, LICENSE, Makefile) - untracked. Как опенсорс это
  пока пустышка.
- CLA обязателен по тексту LICENSE, но самого текста CLA не существует -
  для дуал-лицензирования это дыра: принимать вклад нельзя, подписывать нечего.
- CI отсутствует полностью (нет ни .github/, ни аналога для gitverse).

### Плохо

- Makefile: все ключевые цели - обертки над pwsh-скриптами. На Windows нет
  make из коробки, на Linux/macOS нет pwsh. Не работает нигде "из коробки"
  при заявленной кроссплатформенности.
- Cargo.lock в .gitignore (.gitignore:28) - для приложения ошибка: теряется
  воспроизводимость сборок. Должен быть в гите.
- README обещает то, чего нет: src/sync/ (QRESYNC/IDLE), адаптеры ews/jmap,
  Tantivy; ссылки на несуществующий docs/. Опечатка "метаданны" (README:60).
- Тесты: всего 6 (3 добротных на шифрование в storage/mod.rs, 2 в oauth, 1 в
  imap). repo.rs и схема БД не покрыты ни одним тестом. Каталога tests/ нет.
- Нет CONTRIBUTING.md, SECURITY.md, CHANGELOG. Нет rustfmt.toml/clippy.toml.

### Ок

- .env.example: только публичный client_id, секреты класть не провоцирует.
- .gitignore по секретам корректный; токенов в коде нет.

## 8. Приоритеты

### P0 - до любого следующего шага

1. BODY.PEEK[] вместо RFC822 (imap.rs:244) - сейчас клиент портит флаги всего
   ящика при первом же запуске.
2. XSS в палитре поиска и вложениях (app.js:196, 567) - эксплуатируется
   контентом входящего письма, дает доступ ко всем Tauri-командам.
3. Закоммитить код + Cargo.lock; полный официальный текст AGPL-3.0 + файл CLA.
4. Blob-имена: keyed BLAKE3 или случайные id вместо FNV(plaintext).
5. Ключи: подмешивать OsRng к мышиной энтропии через HKDF.

### P1 - минимум жизнеспособного клиента

6. SMTP-отправка (lettre) + композер: отправка, ответ с цитированием и
   адресатом, автосохранение черновика, реальные вложения.
7. HTML-рендер писем в sandboxed iframe + санитайзер + блокировка удаленных
   изображений с per-sender разблокировкой (has_remote_content уже считается).
8. Инкрементальный sync (UIDVALIDITY/UIDNEXT уже в схеме) + outbox-воркер с
   полноценным payload + IDLE/периодический sync + refresh OAuth-токена.
9. Миграция 0011: UNIQUE на contacts/events/calendars/threads; составной индекс
   (folder_id, date DESC); recursive_triggers или FTS-триггеры на каскады.
10. Убрать паники в main() - показывать диалог ошибки.
11. Fallback на системный браузер для OAuth.

### P2 - продукт, а не прототип

12. Виртуализация списка писем + курсорная подгрузка.
13. Убрать или подписать все плацебо-контролы - сейчас UI систематически врет.
14. Множественное выделение, undo, реальное удаление/архив через outbox.
15. Локализация UI через ftl (сейчас фикция), условия умных папок - по ключам,
    а не русским строкам.
16. Доступность: button вместо div, focus-стили, role/aria.

    не добавить.
18. App lock переделать (Argon2id, проверка в ядре) или убрать.
19. Тесты на repo.rs и миграции; CI (fmt + clippy + test).
20. Кроссплатформенная сборка: заменить pwsh-обвязку (just / cargo xtask).
